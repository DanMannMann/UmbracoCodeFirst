Data Type Interfaces

Code-First data types always use the underlying storage type (DateTime, int or string) for serialisation and deserialisation. Umbraco's default behaviour of returning a converted object or converting all values into strings is bypassed and the underlying data type is passed in and out of the data type converter. Therefore there are only 4 interfaces for implementing a data type.

Data Type Interfaces

  • IUmbracoDateDataType

    Specifies that the data value should be stored as a DateTime2 type. Converter input & output type is SystemDateTime

  • IUmbracoIntegerDataType

    Specifies that the data value should be stored as a 32-bit integer type. Converter input & output type is SystemInt32

  • IUmbracoNtextDataType

    Specifies that the data value should be stored as an Ntext type. Converter input & output type is SystemString

  • IUmbracoNvarcharDataType

    Specifies that the data value should be stored as an Nvarchar(MAX) type. Converter input & output type is SystemString

There are generic interfaces, inherited by the 4 above, which allow the converter input type to be specified. These may be used in the future to tighten integration with Umbraco's property editor value converters. However, for now, only the constructed generic types specified by the interfaces above (IUmbracoDataType<DateTime>, IUmbracoDataType<string> & IUmbracoDataType<int>) are supported, as the code-first core only ever accesses the unconverted data value.

 Caution

If you match up the wrong interface to a data type it is likely to cause runtime exceptions when rendering the document using strongly-typed models.

Almost all property editors use a string, where any which store a lot of data (e.g. Grid) tend to store Ntext and those with shorter values (e.g. textstring) tend to store Nvarchar. Information on the exact type for each editor is best obtained by combining a debugger with a cup of tea (or reviewing the built-in data type implementations in code-first).
